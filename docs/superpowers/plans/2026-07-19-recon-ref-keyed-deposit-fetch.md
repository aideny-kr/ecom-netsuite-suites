# Recon — Order-Ref-Keyed Deposit Fetch (kill the windowing artifact) — Plan

> Operator requirements (2026-07-19): "date doesn't really matter as much — reasonable timeframe; sometimes off by weeks" + "order number is an important indicator; utilize the keys and dimensions of the data." Empirical basis (Framework, session ledger 2026-07-19): 47,065 ref-matched pairs — lag p50 −3d / p99 +15d / 147 pairs >28d; **1,089/1,095 (99.5%) of the latest run's `missing_in_netsuite` charges have their counterpart in `netsuite_postings` outside the run's ±14d fetch buffer.**

**Goal:** the order reference — the strongest key in the data — actually decides matching. Deposit candidates are fetched **by the run's charge order-reference set** (date-independent, generous sanity cap), so cross-window pairs match deterministically. Expected effect: `missing_in_netsuite` collapses from ~1,095/run to the genuine handful; fresh-window explained rate rises accordingly.

**Architecture (engine-scoped, deliberately minimal):**
- `order_recon_job._fetch_deposits` gains a second, ref-keyed pass: after `_fetch_charges`, collect the distinct non-null `order_reference`s; fetch `NetsuitePosting` rows whose extracted ref ∈ that set with ONLY a sanity bound `transaction_date >= date_from - REF_MATCH_SANITY_DAYS AND <= date_to + REF_MATCH_SANITY_DAYS` (`REF_MATCH_SANITY_DAYS = 90`, module constant, comment citing the measured p99=15d + >28d tail + operator "weeks" guidance). Union (dedupe by id) with the existing windowed fetch (which still supplies fuzzy-tier candidates for no-ref charges). Ref extraction on the postings side: match on `related_payout_id` containing the ref — implement the ref-keyed query chunked (5000 refs per `IN`/`LIKE ANY`-style batch; check how `related_payout_id` stores the ref — deposit sync writes the extracted sales-order ref there; if it's stored raw (e.g. "Sales Order #R…"), use `substring(related_payout_id FROM 'R[0-9]{9}') IN chunk` or per-tenant pattern via `load_order_ref_pattern` — mirror whatever `_fetch_deposits` already does to derive `order_reference` for matching, DON'T invent a second extraction).
- Everything downstream is unchanged: tier-1 ref equality, consumed-once semantics, tier-2 fuzzy (its ±date proximity scoring untouched — it only serves no-ref residue), classifier, planner. Cross-run double-match risk is structurally low (ref is order-unique; each charge appears in exactly one window by arrival date) — state this in a comment; the planner's cross-run guard remains the backstop.
- **Perf note:** `related_payout_id` LIKE/substring over 26k+ rows per chunk — check for an index; if the query plan is ugly at Framework scale, an expression index migration (`substring(related_payout_id FROM 'R[0-9]{9}')`) is IN SCOPE as migration 090 (additive).
