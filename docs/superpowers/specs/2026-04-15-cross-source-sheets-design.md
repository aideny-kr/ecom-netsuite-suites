## Problem Statement

AI-den connects to NetSuite (transactional ERP) and BigQuery (analytics/marketing). Users ask questions that span both systems — "which SKUs sell best on Shopify (BQ) but have the lowest margin in NetSuite?" — and the agent currently handles one source per question via disambiguation. Users also want query results exported to Google Sheets for sharing, further analysis, and integration with existing finance/ops workflows.

## Demand Evidence

- Framework team needs ad-hoc cross-referencing between NetSuite (live SOs, POs, inventory) and BigQuery (Shopify/Heap analytics) — not one killer query, but the ability to pull from both sources in the same answer
- Google Sheets export is the most-requested output format for sharing results with people who don't have AI-den access
- "Agent writes to Sheets" is the specific use case: query data, create a formatted Sheet, return the URL

## Status Quo

- Users ask a question, the agent picks one source (NetSuite or BigQuery), and answers from that source only
- Source pin (session affinity) means follow-up questions stay on the same source
- Cross-source questions get the disambiguation instruction ("I can check NetSuite or BigQuery — which would you prefer?") instead of checking both
- For Sheets, users manually copy data from the chat UI into spreadsheets

## Target User & Narrowest Wedge

**Target:** Framework ops/finance team members who ask analytical questions spanning multiple systems.

**Narrowest wedge:** The agent calls both `netsuite_suiteql` and `bigquery_sql` in the same turn when the question requires it, and can write results to a formatted Google Sheet via a Sheets connector.

## Constraints

- No new routing infrastructure (v2.0 just deleted 6,600 lines of routing)
- Must follow the connector + knowledge profile pattern (same as BigQuery)
- Google Sheets OAuth requires consent flow + encrypted credential storage
- Agent must return the Sheet URL to the user after writing

## Premises

1. Cross-source queries MAY already work with v2.0 since both tool sets are visible to the unified agent. However, the disambiguation instruction actively steers the agent to ask "which source?" rather than calling both. Phase 1 tests will determine if this is a prompt fix or an infrastructure gap. Must test before building anything new.
2. Sheets tools are Python functions using `google-api-python-client` (matching the BigQuery pattern exactly — in-process, async, registered in tool registry). NOT an external MCP server.
3. Sheets auth uses a service account (not user-delegated OAuth) for the wedge. Scopes: `spreadsheets` + `drive.file`. Service account avoids per-user consent flow, Google review, and token-per-user complexity. Sheets are owned by the service account and shared with the user via `permissions.create`.
4. "Agent writes to Sheets" is the wedge — not living sync, not Sheets-as-RAG. Initial tools: `sheets_create` and `sheets_write_range` only. Formatting (`sheets_format_range`) and reading (`sheets_read_range`) deferred to Phase 2.
5. Order: (a) test cross-source as-is, (b) add Sheets connector + profile, (c) test agent writing to Sheets.

## Cross-Model Perspective

Independent cold read from a Claude subagent reinforced all four premises. Key additions:

- **"Data fabric" vision:** Sheets as bidirectional — humans set structure (budget targets, forecast templates), agent enriches with live data from NetSuite + BigQuery. This is Phase 2.

- **50% exists:** mcp-google-sheets (open source) gives the Sheets I/O layer. The 50% to build: connector lifecycle (OAuth, credentials, health check), result-to-Sheet formatting (knowledge profile), cross-source synthesis (prompt engineering).

## Approaches Considered

### Approach A: Test-First Minimal (CHOSEN)
Test cross-source queries on staging as-is. If they work, ship with zero code changes. Add Sheets as a connector (OAuth + 4 tools + knowledge profile). Ship the wedge: agent writes query results to formatted Sheets.
- **Effort:** S-M
- **Risk:** Low
- **Pros:** Validates cross-source for free, ships Sheets wedge fast, builds on proven connector pattern
- **Cons:** Doesn't address Sheets-as-RAG or bidirectional sync yet
- **Reuses:** mcp_connectors table, knowledge profile loader, tool_inventory, BigQuery connector pattern

### Approach B: Full Data Fabric
Same as A, plus Sheets-as-RAG, bidirectional sync, Sheet-driven queries. The "three systems, one brain" vision.
- **Effort:** L-XL
- **Risk:** Medium
- **Pros:** Full competitive moat (Gemini/Claude add-ons can't reach private NetSuite)
- **Cons:** Scope creep, untested demand for bidirectional features

### Approach C: Sheets-First Only
Just the Sheets connector. Skip cross-source testing/optimization. Let cross-source happen organically.
- **Effort:** S
- **Risk:** Low
- **Pros:** Narrowest possible scope
- **Cons:** Misses the cross-source validation that's free to do

## Recommended Approach

**Approach A: Test-First Minimal.** Test cross-source first (free validation), then add Sheets connector following the BigQuery pattern. Ship the write wedge.

### Phase 1: Validate Cross-Source (zero code changes)

Run 5 cross-source queries on staging with the Framework tenant:
1. "What's our top-selling SKU by BigQuery revenue that has the lowest gross margin in NetSuite?"
2. "Compare BigQuery session counts by product category against NetSuite sales order counts for last month"
3. "Which marketing channels in BigQuery drove orders that had the highest return rate in NetSuite?"
4. "Show me customer acquisition cost from BigQuery alongside lifetime order value from NetSuite for top 10 customers"
5. "Average time between first website visit (BigQuery) and first purchase (NetSuite) by product line"

Record: did the agent call both tools? Did it join results correctly? Where did it fail?

If 3+ work without prompt changes: cross-source is a free feature. Add a `cross_source.yaml` knowledge profile with join-key guidance and ship.

If <3 work (likely — the disambiguation instruction says "ask the user which source"): update the disambiguation instruction in `prompt_assembler.py` from "If genuinely unsure, ask the user" to "If the question clearly requires both sources, call both tools. Only ask the user when you genuinely cannot determine which source is authoritative." Then retest. Also handle `source_pin` for cross-source sessions: when both tools are called in a turn, `_compute_source_pin_update` clears the pin to `None`. Consider adding `source_pin = "cross"` to signal cross-source mode for follow-ups, or simply let `None` with the updated disambiguation instruction handle it naturally.

### Phase 2: Google Sheets Connector

**Backend:**
1. Implement 2 tools in Python using `google-api-python-client` (same pattern as `bigquery_tools.py`): `sheets_create` and `sheets_write_range`. Register in tool registry. No external MCP server — all in-process.
2. Google service account auth (scope: `spreadsheets`). Service account JSON key stored encrypted in `mcp_connectors` with `provider: "google_sheets"`. Same pattern as BigQuery exactly.
3. After writing data, share the Sheet with the requesting user's email via Sheets API `permissions.create` (role: writer). Return the Sheet URL.
4. Health check: verify service account can list recent Sheets (same pattern as BigQuery's `datasets.list`).

**Knowledge Profile:** `google_sheets.yaml`
```yaml
profile_id: google_sheets
display_name: "Google Sheets"
trigger_tools:
  - sheets_create
  - sheets_write_range
prompt_fragment: |
  ## Google Sheets Context

  You can create Google Sheets and write data to them. When the user asks to export data to a Sheet:
  1. First query the data (SuiteQL, BigQuery, etc.)
  2. Call sheets_create with a descriptive title
  3. Call sheets_write_range with headers in row 1 followed by data rows
  4. The Sheet URL will be returned automatically — present it to the user

  Data format: pass a 2D array where row 0 is headers. Numbers as numbers, not strings.
  If the query returned >1000 rows, warn the user before writing.
  If sheets_write_range fails, tell the user and offer to retry or download as CSV instead.
rag_partitions: []
```

**Frontend:**
- Settings: Google Sheets connector card (copy BigQuery connector card pattern)
- Chat: `sheets_link` SSE event type for clickable Sheet URL card (similar to `data_table` event)

### Phase 3: Cross-Source-to-Sheets Demo

The end-to-end flow: "Compare NetSuite revenue vs BigQuery marketing spend by channel and put it in a Google Sheet." Three systems, one sentence, formatted output with a clickable URL.

## Open Questions

1. ~~Service account vs. user-delegated OAuth~~ RESOLVED: Service account. Sheets owned by service account, shared with user via permissions API.
2. Should cross-source results be combined in a single `data_table` SSE event or rendered as two separate tables? (Likely: let the agent present them naturally, two tool calls = two `data_table` events)
3. Rate limits: Google Sheets API has 300 req/min per project. Single `sheets_write_range` for <1000 rows fits in one call (~500KB). Sufficient for initial usage.
4. Should there be an explicit "Export to Sheets" button in the chat UI, or should the agent decide when to offer Sheets export? (Start with agent-only, add button if users ask for it)

## Success Criteria

- Cross-source validation: 3+ out of 5 test queries work without code changes
- Sheets connector: user can ask "put this in a Google Sheet" and get a formatted Sheet URL back
- Demo: three-system flow (NetSuite + BigQuery → Sheets) works end-to-end
- No new routing infrastructure, no new agent types

## Distribution Plan

Existing deployment pipeline covers this. No new artifacts to distribute. The Sheets connector is backend-only (connector + tools + profile). Frontend change is one new SSE event card.

## Dependencies

- Google Cloud project with Sheets API + Drive API enabled (Framework already has `framework-marketplace` GCP project for BigQuery)
- Service account JSON key with scopes: `spreadsheets` + `drive.file` (drive.file needed for `permissions.create` to share Sheets with users)
- `google-api-python-client` + `google-auth` added to `pyproject.toml` (synchronous SDK, wrap calls with `asyncio.to_thread()`)
- User email lookup: `sheets_write_range` tool needs the user's email to share the Sheet. Pass `user_email` in tool context (look up from `actor_id` in `execute_tool_call`) or make sharing a separate `sheets_share` tool call
