Stage 1 — Input Processing
💬
User Message Intake
Sanitization, history loading, and RAG pre-retrieval
🛡️ Input Sanitizer
Strips prompt injection tags (<system>, <instructions>, <tool_call>) and dangerous control characters. Also sanitizes assistant output via _sanitize_assistant_text() to strip leaked tool-call XML.
⚡ Chitchat Short-Circuit
Regex detects greetings, compliments, and affirmations ("thanks", "awesome", "hi"). Short-circuits expensive context assembly — skips entity resolution, domain knowledge, and pattern retrieval for conversational messages.
📜 History Loader
Loads full conversation history from PostgreSQL. If >12 messages, compacts old turns into a dense summary via fast LLM call while keeping last 4 messages verbatim.
📚 RAG Pre-Retrieval
Embeds the user question (OpenAI text-embedding-3-small, 1024-dim) and retrieves top-K similar doc chunks via pgvector cosine similarity. Injected as <context> block.
Stage 2 — Context Assembly (Concurrent)
🧠
Parallel Context Enrichment
Three knowledge sources queried concurrently via asyncio.gather
NER Entity Resolution
Fast LLM call (Haiku) extracts business entities from the user message → pg_trgm fuzzy matches against 2,109 tenant entity mappings → produces <tenant_vernacular> XML block mapping names to NetSuite script IDs.
EMBED Domain Knowledge
Embeds user query → pgvector cosine search over curated domain knowledge chunks (SuiteQL syntax rules, NetSuite data model patterns, field-type reference). Results injected as <domain_knowledge>.
MATCH Proven Patterns
Finds previously successful SuiteQL queries for similar questions via embedding similarity. Injects them as <proven_patterns> — the agent learns from its own past successes.
Stage 3 — System Prompt Construction
⚙️
Dynamic Prompt Assembly
Tenant-specific system prompt built from 8 layered sources
📋 Base Template
Active system prompt template from prompt_template_service. Includes SuiteQL rules, tool guidance, output formatting, and ROWNUM pagination directives.
🏢 Tenant Profile
Industry, subsidiaries, chart of accounts, item types, fiscal calendar, custom segments — all discovered via metadata pipeline.
🔧 Custom Fields
Discovered custbody_*, custcol_*, custitem_*, custrecord_* fields with their labels and types. Up to 60 fields per section.
🪪 Brand Identity
Injects the tenant's brand name so the AI knows its own identity. When asked "what's your name?" the agent responds with the white-labeled brand, not "Claude" or "Suite Studio AI".
🎭 Soul Config
Tenant-specific AI tone (formal vs. casual) and NetSuite quirks from soul.md. Shapes how the agent communicates and handles edge cases.
🛠️ Tool Inventory
Dynamic list of all available tools (local + external MCP). If external SuiteQL MCP tools exist, the prompt guides the agent to prefer them over local REST.
📁 Workspace Context
If the user is in an IDE workspace: file listing (up to 50 files), workspace ID, and instructions for file operations and viewing context.
Stage 4 — Agent Execution
🤖
Unified Agent Execution
Single agent with 10 tools, up to 10 reasoning steps per turn
A single unified agent receives all context blocks and the full tool inventory. The LLM decides which tools to call based on the user's question — no explicit routing step needed. Streams tokens via SSE as they're generated. Financial queries are automatically detected and augmented with accounting-specific SuiteQL rules (TAL joins, CONSOLIDATE functions, account type filters).
PRIMARY Unified Agent
Single agent with all 10 tools and max 10 steps. Context blocks (vernacular, domain knowledge, proven patterns) injected into system prompt. Anti-hallucination guard: forbidden from guessing custom field IDs without verification.
AUTO-DETECT Financial Report Mode
Intent classifier detects financial queries (P&L, Balance Sheet, GL). Augments the task with verified column names, mandatory filters (posting=T, primary book), currency consolidation rules, and accounting period joins. Domain knowledge top-K boosted to 5.
FALLBACK Multi-Agent Coordinator
Legacy path available per tenant flag. Heuristic regex classifier → specialist agent dispatch → result synthesis. Routes: SuiteQL, RAG, Analysis, Workspace.
Tool Inventory (10 Tools)
🔍 netsuite_suiteql
Execute SuiteQL queries against tenant's NetSuite via REST API (OAuth 2.0). Read-only validation, row limit enforcement, pagination, and permission diagnostics for empty results.
📋 netsuite_get_metadata
Retrieve discovered custom fields, records, lists, scripts, and org structure. Used by the agent to verify field names before constructing queries.
📖 rag_search
Vector search over 3,198 embedded doc chunks. Returns relevant NetSuite documentation, SuiteQL syntax guides, and platform knowledge.
🌐 web_search
Fallback web search for current information not in the doc store. Rate-limited to 10 req/min with 15s timeout.
💻 workspace_* (4 tools)
list_files, read_file, search, propose_patch — IDE file operations with auto-injected workspace ID.
💡 tenant_save_learned_rule
Persists user corrections and preferences as tenant-specific rules. Available for future sessions across all agents.
Stage 5 — Post-Processing & Learning
✨
Response Delivery & Memory Update
Streaming output, audit logging, billing, and background learning
📡 SSE Streaming
Tokens stream to the frontend in real-time via Server-Sent Events. Three event types: text (tokens), tool_status (progress), message (final result). Output sanitized to strip any leaked tool-call XML before delivery.
📝 Audit & Billing
Every turn logged with: mode, provider, model, step count, token usage, tools called, doc chunks used. Credits deducted for non-BYOK tenants.
🧩 Memory Updater
Fire-and-forget background task. Regex-gated (zero cost for 95% of messages). When user says "actually..." or "remember that...", extracts correction → persists as TenantLearnedRule.
📈 Pattern Extraction
After successful SuiteQL queries, extracts the SQL pattern, embeds the user question, and upserts into tenant_query_patterns — building a self-improving query library.

How Knowledge Flows Into Every Chat Turn
Every user message triggers a concurrent knowledge assembly pipeline. Entity Resolution, Domain Knowledge, and Proven Patterns run in parallel via asyncio.gather(), injecting their results as XML context blocks into the agent's system prompt. Meanwhile, RAG doc chunks are pre-retrieved and injected into the user message itself. The Tenant Metadata and Soul Config are loaded at prompt construction time. Learned Rules feed back into Entity Mappings and are available for the next session. The result: the agent starts every turn with the full context of who this tenant is, what their NetSuite looks like, what's worked before, and what corrections the user has made.

🤖
Unified Agent Execution STREAMING
The agent reasons in a <reasoning> block, plans the query approach, then calls netsuite_suiteql with:
SELECT c.companyname, SUM(t.amount) AS total_revenue
FROM transaction t JOIN customer c ON t.entity = c.id
WHERE t.type = 'CustInvc' AND t.trandate >= TRUNC(ADD_MONTHS(SYSDATE, -1), 'MONTH')
GROUP BY c.companyname ORDER BY total_revenue DESC
WHERE ROWNUM <= 10
UnifiedAgent.run_streaming() netsuite_suiteql SSE text events
🌐
NetSuite REST API Call
The netsuite_suiteql tool validates the query, enforces row limits, then sends it to NetSuite via REST API (/services/rest/query/v1/suiteql) with OAuth 2.0 Bearer token. Token auto-refreshes if expired. Results returned as JSON.
